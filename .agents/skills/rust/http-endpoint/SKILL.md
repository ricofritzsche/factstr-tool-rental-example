---
name: rust-http-endpoint
description: Wire an existing Rust feature slice to an Axum HTTP endpoint without moving business behavior into HTTP.
---

# Purpose

Use this skill when adding an HTTP endpoint for an already implemented Rust feature slice in this repository.

HTTP wiring is a delivery boundary.

The endpoint should translate:

- HTTP JSON request
- feature request type
- feature result
- feature error
- HTTP response

It must not implement domain behavior.

# Core Rule

Keep feature behavior inside:

```text
rust/src/features/<feature_name>/
````

Keep HTTP translation inside:

```text
rust/src/http/<feature_name>.rs
```

Keep the route composition inside:

```text
rust/src/routes.rs
```

Do not put Axum types inside feature slices.

# Required Location

For every feature endpoint, add or update:

```text
rust/src/http/
  mod.rs
  <feature_name>.rs
```

Example:

```text
rust/src/http/register_tool.rs
```

The HTTP file owns only:

* Axum handler function
* HTTP request extraction
* HTTP response mapping
* HTTP error mapping
* small endpoint-specific HTTP DTOs if needed

# Feature Boundary

The feature slice remains independent of HTTP.

Allowed inside feature slice:

* feature request type
* feature response type
* feature error type
* process function
* feature-local context loading
* feature-local pure decision logic
* feature-local FACTSTR append/query shell code

Forbidden inside feature slice:

* Axum extractors
* HTTP status codes
* JSON response wrappers
* route definitions
* HTTP error response types

# HTTP Handler Responsibilities

The handler should:

1. receive application state/store from the existing app state wiring
2. deserialize JSON into the feature request type
3. call the feature process function
4. map success to the documented HTTP status and JSON body
5. map feature errors to stable HTTP status codes and stable JSON error codes

The handler must stay thin.

It must not:

* validate business rules beyond request deserialization
* decide domain behavior
* query FACTSTR directly if the feature already owns that behavior
* append facts directly
* build projections
* introduce service/repository/manager layers

# Route Responsibilities

`rust/src/routes.rs` owns route composition only.

It may register routes such as:

```text
POST /tools
GET /tools
POST /tools/{tool_id}/checkout
POST /tools/{tool_id}/return
```

It should not contain feature behavior or error mapping logic.

# Error Response Shape

Use a small stable JSON error shape unless the feature spec says otherwise:

```json
{
  "code": "..."
}
```

Do not expose raw FACTSTR errors, database errors, stack traces, or internal diagnostic strings in HTTP response bodies.

# Error Mapping Rules

Map feature-level errors to HTTP status codes explicitly in the HTTP file.

Typical mapping:

```text
validation error              -> 400 Bad Request
duplicate/conflict error      -> 409 Conflict
not-found domain error        -> 404 Not Found
invalid state transition      -> 409 Conflict
store_error / unexpected IO   -> 500 Internal Server Error
```

Use stable feature error codes from the feature implementation.

Do not invent new HTTP-only domain meanings.

# Request Rules

Use the existing feature request shape unless the API spec explicitly defines a separate HTTP request DTO.

A request must not accept system-generated fields.

Examples:

* `tool_id` generated by the system must not be accepted in `POST /tools`
* commit metadata such as sequence number or occurred-at must not be accepted from HTTP clients

If the feature request uses `#[serde(deny_unknown_fields)]`, preserve that behavior in HTTP tests.

# Success Response Rules

Use the existing feature response shape unless the API spec explicitly defines a separate HTTP response DTO.

Keep responses narrow.

Do not return projections or expanded state from command endpoints unless the feature/API spec requires it.

# State Access

Use the existing application state/store wiring.

Do not introduce:

* service layer
* repository layer
* manager object
* generic application framework
* global mutable singleton

If testing needs state construction, create the smallest local test setup.

# Testing Rules

Add endpoint tests for each HTTP endpoint.

Tests should exercise the route or handler through HTTP-level behavior, not only call the feature function.

Preferred proof points:

* correct route and method
* a valid request returns the expected status
* success JSON body has the expected shape
* generated fields are returned but not accepted from caller input
* validation errors map to stable error codes and statuses
* conflict/domain errors map to stable error codes and statuses
* store/internal errors map to stable error code and status when this can be tested with a small local seam
* the endpoint does not append or change the state on rejected requests, when this can be inspected cleanly

Use the FACTSTR memory store for HTTP tests when possible.

Do not require PostgreSQL for HTTP endpoint tests unless the endpoint specifically tests PostgreSQL wiring.

Do not introduce a broad mock framework.

Small local test seams are acceptable when needed to prove error mapping.

# Documentation Rules

Update `rust/README.md` with a short endpoint note.

Include only:

* method and path
* success status
* success body shape at a high level
* important domain/status mappings

Do not turn the README into a tutorial.

Do not document features that are not implemented.

# Dependency Rules

Do not change FACTSTR dependency versions.

Do not introduce git dependencies for FACTSTR.

Add endpoint-test dependencies only when necessary and keep them local/minimal.

Do not add unrelated web framework packages beyond the existing Axum stack.

# Forbidden Structure

Do not create folders or modules named:

```text
domain
shared
common
core
utils
helpers
services
managers
repositories
models
entities
adapters
controllers
```

Use names that describe ownership directly.

`http` is allowed because it owns the HTTP delivery boundary.

# What To Avoid

Do not:

* move business behavior from features into HTTP handlers
* call FACTSTR directly from HTTP for behavior already owned by a feature
* add generic error frameworks
* add central API response abstractions prematurely
* add service/repository wrappers around feature calls
* broaden the route file into a behavior file
* rewrite bootstrap code as part of endpoint wiring

# Default Implementation Steps

1. Read the feature spec.
2. Inspect the existing feature slice.
3. Inspect current app state and route wiring.
4. Add `rust/src/http/mod.rs` if missing.
5. Add `rust/src/http/<feature_name>.rs`.
6. Register the route in `rust/src/routes.rs`.
7. Add HTTP tests.
8. Add a short README note.
9. Run verification.

# Verification

Run from `rust/`:

```bash
cargo fmt --check
cargo check
cargo clippy -- -D warnings
cargo test
```

# Output Expected From Codex

After implementation, summarize only:

1. files changed
2. endpoint added
3. error mapping added
4. tests added
5. dependency changes
6. verification results
7. remaining intentional limitations
